# AutoHeal On-Box Runbook

This README explains how to:

- Run AutoHeal locally (non-SLM pipeline)
- Verify accuracy and performance gates
- Deploy on a Windows on-box device
- Verify the same gates after deployment

What this project does:
- Loads daily, pre-aggregated sensor CSV data.
- Runs per-sensor anomaly detection.
- Correlates anomalies into incidents.
- Composes deterministic remediation/verification plans and uses on-device SLM passes (Ollama) to produce grounded, actionable diagnosis outputs.

## Scope

This repo includes detectors, correlation, diagnosis, **SLM integration** (e.g. Ollama, demo UI), and other features. This runbook focuses on the **non-SLM** path and on-box deployment.

The path covered here is:

`collectors -> detectors -> accuracy gate -> correlate -> diagnose`

SLM is optional for this runbook; see `scripts/README.md` and related scripts for SLM and other flows.

## KB-Grounded Diagnosis (RAG)
- Runtime diagnosis now retrieves context from PDF knowledge base files in `data/KB`.
- Retrieval is filtered for laptop-relevant fault domains and suppresses unrelated server/network guidance by default.
- Runtime generation asks the SLM for a full plan (`intent`, `steps`, `verify`, `risk`, `diagnosis_text`, `sources`) grounded in retrieved KB context.
- A second-pass actionability refiner (`llama3.2:latest` by default) can rewrite `steps` and `verify` using full extracted text from selected source PDFs.
- `steps` and `verify` must include inline citations tied to retrieved KB pages; invalid/non-retrieved citations are rejected.
- Deterministic playbook + KB extraction remains the fallback if grounded SLM steps/verify fail validation.
- Runtime post-processing normalizes common refiner citation formats before grounding validation (for example, `allowed source:` wrappers and `source.pdf:pN` placeholders).
- Prompt trimming may occur for token budget safety; canonical RAG retrieval hits are still preserved for grounding and finalization.
- Refined `diagnosis_text` must mention the selected fault domain/subsystem; off-domain text is rejected.
- If grounded plan validation fails, deterministic plan fields remain unchanged.

Tune in `config/config.yaml` under `diagnosis`:
- `slm_rag_enabled`
- `slm_rag_kb_dir`
- `slm_rag_top_k`
- `slm_rag_min_score`
- `slm_rag_chunk_chars`
- `slm_rag_max_context_chars`
- `slm_actionability_refiner_enabled`
- `slm_actionability_refiner_model`
- `slm_actionability_refiner_system_prompt`
- `slm_actionability_refiner_options`
- `slm_actionability_refiner_timeout_sec`
- `slm_actionability_refiner_keep_alive`
- `slm_actionability_refiner_url`
- `slm_actionability_refiner_max_source_chars`
- `slm_mock_remediation_enabled`
- `slm_mock_remediation_outcomes`
- `slm_mock_remediation_domain_actions`
- `slm_playbook_path`

Observability outputs:
- `out/diagnostics.jsonl` includes `rag_hit_count`, `rag_sources`, `rag_top_hits`, `deterministic_primary`, `slm_text_refined`, `run_path_flags`, `derived_plan_reasons`, `fallback_used`, and `derived_plan_used`.
- When actionability refiner is enabled, rows also include `actionability_refiner_attempted`, `actionability_refiner_used`, `actionability_refiner_model_used`, `refined_steps_used`, `refined_verify_used`, `source_docs_full_chars`, `source_docs_returned_chars`, and truncation fields.
- When mock remediation is enabled, rows include `mock_remediation_enabled`, `mock_remediation_domain`, `mock_remediation_action_count`, `mock_remediation_overall_success`, and `mock_remediation_actions`.
- `deterministic_primary: true` means plan composition stayed in deterministic primary mode.
- `slm_text_refined: false` means deterministic diagnosis text seed was kept.
- `out/diagnostics_debug/*.json` retains full `rag_hits`, selected source documents context stats, first-pass and actionability-pass prompts, and per-attempt traces.

## 6) Deploy on-box (Windows services)

The on-box deployment uses NSSM services and named pipes.

### Prerequisites

- Windows machine with admin rights
- Python installed
- `nssm.exe` available

### Install services

```powershell
.\services\install_services.ps1 -NssmPath "C:\Tools\nssm.exe"
```

What this does:

- Copies repo to `C:\AutoHealOnBox` (excluding `.venv`)
- Creates `C:\AutoHealOnBox\.venv` if missing
- Installs dependencies from `requirements.txt` (unless `-SkipPipInstall`)
- Installs one detector service per sensor: `AutoHeal_<sensor>`
- Installs correlation service: `AutoHeal_Correlation`
- Configures service logs in `C:\AutoHealOnBox\out\service_logs`

Optional installer flags:

```powershell
.\services\install_services.ps1 -NssmPath "C:\Tools\nssm.exe" -CleanInstallDir
.\services\install_services.ps1 -NssmPath "C:\Tools\nssm.exe" -SkipPipInstall
```

### Verify services are running

## 8) Troubleshooting

- `accuracy_gate` fails:
  - Rebuild labeled splits with `build_labeled_holdout.py`
  - Check `out/metrics/sensor_accuracy.csv` for `eligible` and `status` columns
- Missing telemetry rows:
  - Confirm `out/metrics/pipeline_telemetry.csv` is writable
  - Re-run profile with explicit `--run_id`
- Service not starting:
  - Check `Get-Service AutoHeal_*`
  - Inspect `C:\AutoHealOnBox\out\service_logs\*.err.log`
- No incidents generated:
  - Ensure prediction files are being written in `out/predictions/service`
  - Ensure correlation service is running

# Scripts Usage

All commands assume you run them from the repo root:

```bash
python scripts/<script>.py [options]
```

If you are using the virtual environment, run:

```bash
.\.venv\Scripts\python scripts/<script>.py [options]
```

## Script intent at a glance

- Production pipeline flow:
  - `run_detect.py` -> `run_correlate.py` -> `run_slm_diagnose.py`
  - or all-in-one: `run_pipeline.py`
- Model-development flow:
  - Generate debug traces with `run_slm_diagnose.py --debug`
  - Evaluate gate metrics with `eval_slm_quantized.py`
  - Build SFT data with `build_slm_finetune_dataset.py`
  - Train/export under `training/slm/`

## Core pipelines

- `run_detect.py`  
  Sequentially runs all detectors and writes predictions to `out/predictions`.  
  Example:
  ```bash
  python scripts/run_detect.py --threshold 0.5 --sample_seconds 0.5
  ```

- `run_detect_parallel.py`  
  Runs detectors in parallel; supports service-like looping.  
  Examples:
  ```bash
  python scripts/run_detect_parallel.py --max_workers 8
  python scripts/run_detect_parallel.py --loop --sleep_seconds 300
  ```

- `run_correlate.py`  
  Correlates anomalies from `out/predictions` into `out/incidents.jsonl`.  
  Examples:
  ```bash
  python scripts/run_correlate.py
  python scripts/run_correlate.py --include_state
  ```

- `run_pipeline.py`  
  Runs end-to-end Detect -> Correlate -> SLM Diagnose with demo-parity defaults.
  - Requires `--device` by default (use `--allow_all_devices` for legacy all-device runs).
  - `--pipeline_type ollama` uses demo Ollama defaults.
  - `--pipeline_type onnx` defaults both primary and refiner models to `models/onnx`.
  Examples:
  ```bash
  python scripts/run_pipeline.py --data_dir data --out_dir out --config config/config.yaml --pipeline_type ollama --device <device_id> --debug_slm
  python scripts/run_pipeline.py --data_dir data --out_dir out --config config/config.yaml --pipeline_type onnx --device <device_id>
  ```

- `generate_slm_prompts.py`  
  Builds `out/incidents_with_priors.csv` and `out/slm_ranked_prompts.csv`.  
  Example:
  ```bash
  python scripts/generate_slm_prompts.py
  ```

- `run_slm_diagnose.py`  
  Runs deterministic-primary diagnosis from `out/incidents.jsonl` with staged SLM refinement and per-call metrics CSV.  
  Uses optional KB-grounded RAG retrieval from `data/KB` (configured in `config/config.yaml`) for laptop-domain remediation/verification grounding.  
  After refinement, it can run a function pipeline for demo flows:
  1) function-selection SLM (chooses callable Python functions)
  2) function execution from `diagnosis/function_registry.py`
  3) post-execution diagnosis SLM (separate output block; primary diagnosis remains unchanged)
  Function outcomes are deterministic and configurable for demos.  
  Can also append legacy mock remediation stage (no side effects).  
  Emits observability metadata (`deterministic_primary`, `slm_text_refined`, `rag_sources`, `rag_top_hits`, `derived_plan_reasons`) in `out/diagnostics.jsonl` and detailed traces in `out/diagnostics_debug/*.json`.  
  Fallback model chains are ignored in deterministic-primary mode.
  Example:
  ```bash
  python scripts/run_slm_diagnose.py --incidents out/incidents.jsonl --slm_model local/llama3.2-1b-diagnose-lora-q8_0:latest --ab_models llama3.2:latest --metrics_out out/diagnostics_metrics.csv
  ```

C) Event-based time series evaluation (simple detection delay)


def events_from_labels(y_true):
    # returns list of (start_idx, end_idx) for each contiguous anomaly block
    events = []
    in_event = False
    for i, v in enumerate(y_true):
        if v == 1 and not in_event:
            start = i; in_event = True
        if v == 0 and in_event:
            events.append((start, i-1)); in_event = False
    if in_event:
        events.append((start, len(y_true)-1))
    return events

def detection_delay(y_true, y_pred):
    events = events_from_labels(y_true)
    delays = []
    for (s,e) in events:
        preds = np.where(np.array(y_pred[s:e+1]) == 1)[0]
        if len(preds) > 0:
            delay = int(preds[0])  # steps after s
            delays.append(delay)
    return {
        "Detected_Events": len(delays),
        "Total_Events": len(events),
        "Event_Recall": len(delays) / len(events) if events else 0.0,
        "Median_Delay": float(np.median(delays)) if delays else None,
        "P90_Delay": float(np.percentile(delays, 90)) if delays else None
    }
